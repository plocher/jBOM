# ADR 0006: Production Folder Convention, ProjectMetadata, Fabrication Service Decomposition, and Diagnostic Collection Principle
Date: 2026-05-08
Status: Accepted
Related: #226, ADR 0005 Phase 3
## Context
ADR 0005 Phase 3 called for "packaging/output and per-project option persistence services."
This ADR records the specific decisions reached during the architectural design for issue #226,
covering: the production folder structure, project metadata as a first-class concept,
fabrication service decomposition (ZipArchiver, GerberPackager, BackupService), and the
diagnostic collect-always / adapters-decide-display principle.

Fabrication Toolkit (FT) was used as a reference implementation. Its production folder
structure and artifact naming conventions were adopted as a baseline rather than invented
from scratch, ensuring jBOM output is familiar to existing KiCad users.
## Decision Drivers
- CLI and plugin adapters must produce consistent, predictable output without duplicating logic.
- Services must expose the capabilities invoked by workflows — no adapter-layer file I/O.
- jBOM and FT can coexist in a KiCad installation without clobbering shared output.
- Diagnostic information must always be available to all adapters (CLI, plugin detail view).
- Fabricator-selection is the user's high-level choice; fiddly flags live in YAML config.
## Production Folder Convention
jBOM's `fab` workflow writes to a `production/` directory in the project root, following FT's
established structure. This ensures user familiarity and permits coexistence.
### Artifact layout
```
production/
  jbom.csv                          ← BOM (jBOM-generated, fabricator-ready)
  cpl.csv                           ← CPL/placement (jBOM-generated)
  {title}_{revision}.zip            ← gerbers archive (fabricator upload artifact)
  backups/
    {title}_{revision}_{timestamp}.zip   ← snapshot of all production artifacts
```
`{title}` comes from the KiCad title block (`GetTitle()`); `{revision}` from `GetRevision()`.
Fallback when title is absent: basename of the `.kicad_pro` file. Spaces normalized to underscores,
special characters removed.

jBOM uses `jbom.csv` and `cpl.csv` (not `bom.csv` / `positions.csv`) to avoid collisions
with FT artifacts when both plugins are installed.

Netlist (`netlist.ipc`) and designators (`designators.csv`) are not produced by default.
Netlist generation is opt-in via `--netlist`. Designators are not in scope (tracked separately).

Intermediate gerber files in a temporary directory are cleaned up after zipping unless
`--debug` is specified.
## ProjectMetadata
`ProjectMetadata` is a first-class frozen dataclass carrying project-release attributes:
- `name: str` — project name (from title block or `.kicad_pro` basename)
- `version: str` — revision string (from title block, may be empty)
- `release_timestamp: datetime` — wall-clock time at workflow invocation

`ProjectMetadata` is read from the KiCad project files and is distinct from artifact naming.
Artifact names are derived from `ProjectMetadata` downstream. Both `SchematicReader` and the
PCB reader must expose title block attributes (`title`, `revision`) to support this.
## Fabrication Service Decomposition
Services expose the capabilities that workflows invoke. The `fab` workflow must not perform
artifact manipulation that isn't exposed through individual service APIs.

### Service hierarchy
```
FabricationWorkflow
  BOMWorkflow         → jbom.csv  (via friend serializer)
  POSWorkflow         → cpl.csv   (via friend serializer)
  GerberExporter      → gerbers/  (kicad-cli subprocess)
  GerberPackager      → {title}_{revision}.zip  (uses ZipArchiver)
  BackupService       → backups/{title}_{revision}_{timestamp}.zip  (uses ZipArchiver)
```
### ZipArchiver
A general-purpose archive primitive:
```
ZipArchiver.archive(source_paths: list[Path], archive_path: Path) -> None
```
Used by both `GerberPackager` and `BackupService`. Has no knowledge of what the files are.
### GerberPackager
Consumes `GerberResult.artifacts` (the list of paths generated by kicad-cli) and delegates
to `ZipArchiver`. Removes the intermediate gerber directory after archiving unless debug mode.
### BackupService
Zips all production artifacts (jbom.csv, cpl.csv, gerbers zip) into a dated backup archive
in `production/backups/`. Uses `ZipArchiver` with explicit `source_paths`.
## Diagnostic Collection Principle
Services **always collect and return all diagnostics** in the result contract.
Adapters decide what to display and when.

This means:
- Services must not gate diagnostic generation on environment variables or adapter-level flags.
- `os.environ.get("JBOM_QUIET")` must not appear in any service module — it is an adapter concern.
- The `quiet` concept (suppressing output) belongs on the adapter, not in `*Request` dataclasses.
- CLI adapter: prints INFO-level diagnostics only when `--verbose` is set.
- Plugin adapter: stores all diagnostics; surfaces them in a detail view on demand.

Consequence: `BOMWorkflow._generate()` removes its `JBOM_QUIET` env-var gate. Resolution
notes ("Found .kicad_pcb, trying to find matching schematic...") are always appended to
`diagnostics`. `POSRequest.quiet` is removed; `POSWorkflow._generate()` removes its
`request.quiet` gate.

Future work: add `Diagnostic(severity, message)` typed entries to replace bare strings,
enabling adapter-side filtering by severity without parsing message text.
## Per-Project Persistence
CLI adapters: read-only consumers of the config file hierarchy (existing mechanism).
No write-back from `jbom` CLI commands.

Plugin adapter (future, #227): persists the high-level fabricator selection (`--jlc`, etc.)
to a per-project `jbom-options.json`, not individual flag values. Individual flag customization
belongs in custom fabricator YAML files. The plugin exposes a config-schema-aware fabricator
selector (dropdown), not FT-style individual checkboxes.
## FT Coexistence
jBOM and FT can coexist in a KiCad installation writing to the same `production/` directory.
Filename differentiation (`jbom.csv` vs `bom.csv`) prevents silent overwrites. Exact
coexistence policy (e.g., subdirectory per tool) is deferred until post-bootstrap validation.
## Consequences
### Positive
- Production output is familiar to FT users.
- Services are testable in isolation; `fab` is a pure coordinator.
- All diagnostic data is available to all adapter types, enabling rich plugin UX.
- Fabricator YAML is the single source of truth for fiddly options.
### Tradeoffs
- Title block reading must be added to both PCB and schematic readers (currently absent).
- BOM/POS file-writing must move out of `cli/fabrication.py` into a service-layer friend.
- `GerberExporter` output (kicad-cli generated paths) is the ZipArchiver input; this is
  an indirect data contract (list of paths) rather than a rich data structure, which is
  acceptable given kicad-cli controls content.
### Out of scope for #226
- designators.csv service (tracked separately)
- netlist as a first-class service (tracked separately)
- Plugin GUI and dialog (tracked in #227)
- Git tag integration for production builds (future enhancement)
- KiCad design file backup bundling (future enhancement)
