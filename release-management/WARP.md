# Release Management Guidelines

## Semantic Release Automation
- **Version Trigger:** Conventional commits (`fix:`, `feat:`, `chore:`, etc.)
- **Automated Actions:** Version bump, git tag, CHANGELOG.md regeneration, PyPI publish.
- **Files Updated:** `pyproject.toml`, `src/jbom/__version__.py`,
  `CHANGELOG.md` (per the documentation charter, `CHANGELOG.md` is
  fully generated; hand-edits are rejected by pre-commit hook + CI
  staleness check — see #305).

## Pre-commit Hooks
Pre-commit hooks auto-fix formatting and check for lint/security/syntax
issues. The iterative re-staging procedure (re-add files after hooks
modify them, repeat until clean) is documented in the `git-workflow`
skill at `.agents/skills/git-workflow/SKILL.md`. Configuration lives
in `.pre-commit-config.yaml`.

## GitHub Secrets Required
- `PYPI_API_TOKEN` — PyPI publishing token.
- `GITHUB_TOKEN` — automatic repository access.

## Release Workflow Rules

- Releases happen on merge to `main`. Developers never push to
  `main` directly; all changes flow through PRs from feature
  branches. See the `git-workflow` skill for the PR procedure.
- Conventional-commit messages on the merged branch drive the version
  bump (`fix:` → patch, `feat:` → minor, breaking change → major).
- `CHANGELOG.md` is regenerated by semantic-release at release time.
  Per the charter clause `CHANGELOG.md` is fully generated, hand-edits
  are not permitted.

## Security
- Token rotation procedures: see incident-response material in this
  directory.
- Use `__token__` as username for PyPI authentication.
- Never expose tokens in logs or commits.
